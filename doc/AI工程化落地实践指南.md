# AI 工程化落地实践指南

> 核心观点：AI Demo 和 AI 产品的差距不在模型能力，而在工程化约束层的厚度。工程化越厚，AI 越可靠；越可靠，才越值钱。

---

## 一、核心架构：三层递进式 AI 落地

传统数字化系统升级不是推倒重来——那风险太大、周期太长、业务方也受不了。正确做法是"给老系统穿衣服"，从外围一层层往里做，每一层都能独立交付、独立产生价值。

```
┌─────────────────────────────────────────────────────┐
│                    第三层：系统缝补                    │
│   跨系统数据搬运 / 格式与语义转换                      │
│   ┌─────────────────────────────────────────────┐    │
│   │              第二层：流程改造                  │    │
│   │   智能审批 / 异常检测 / 非结构化数据处理        │    │
│   │   ┌───────────────────────────────────┐      │    │
│   │   │         第一层：套壳层              │      │    │
│   │   │   自然语言查询 / 对话式操作 /       │      │    │
│   │   │   RAG 知识问答                     │      │    │
│   │   │   ┌─────────────────────────┐     │      │    │
│   │   │   │    传统数字化系统        │     │      │    │
│   │   │   │   (ERP / CRM / OA / ...) │     │      │    │
│   │   │   └─────────────────────────┘     │      │    │
│   │   └───────────────────────────────────┘      │    │
│   └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
```

**三层总览**：

| 层级 | 核心目标 | 改老系统吗 | 风险 | 周期 | 用户感知 |
|------|---------|-----------|------|------|---------|
| 第一层 | 降低使用门槛 | 不改 | 低 | 2-4 周 | 最强 |
| 第二层 | 提升核心效率 | 可能改 | 中 | 1-3 月 | 强 |
| 第三层 | 打通数据孤岛 | 不改 | 中 | 1-2 月 | 中等 |

**落地原则**：从第一层开始做，不动老系统数据库，做只读查询 + 外部写入。AI 输出永远可追溯，决策权在人不在 AI。

---

### 第一层：套壳层（最快见效，ROI 最高）

**一句话理解**：给老系统外面包一层 AI 界面，用户用自然语言跟系统交互，但老系统本身一行代码不改。

几乎所有传统系统的痛点都不是"功能不够"，而是"太难用"——查个报表要选七八个下拉条件、录个单据要点十几下鼠标、新员工培训两周还不会操作。套壳层瞄准的就是这个。

#### 1A. 自然语言查询

**改造前**：财务经理想看"上个月华东区哪个品类卖得最好"，得打开报表系统 → 选日期范围 → 选区域下拉框 → 选品类 → 点查询 → 导出 Excel → 自己排序。操作路径 6-7 步，每次查询 2-3 分钟。

**改造后**：在对话框里打字问"上个月华东区哪个品类卖得最好"，1 秒出结果。

**怎么做的**：用户自然语言 → AI 生成 SQL（注入数据库 Schema + 质检层保障）→ 在只读库执行 → 返回结果表格 + AI 对结果的简要解读。

**关键约束**：
- 只给 AI 数据库只读账号，最坏情况也就是查出个空结果
- 所有 SQL 在执行前必须通过语法、白名单、语义校验
- 敏感字段（工资、身份证号等）在 Schema 注入阶段就过滤掉

#### 1B. 对话式操作

**改造前**：销售员要创建一张报价单，得打开 CRM → 选客户 → 选产品 → 填数量 → 填折扣 → 填有效期 → 点保存。路径长、容易漏填、容易填错。

**改造后**：说一句"帮张三做一个报价单，产品 A 数量 100 单价 50，有效期到月底"，系统自动填好表单预览，人确认一点就提交了。

**怎么做的**：用户自然语言 → AI 提取意图和参数 → 把参数填入一个预生成的表单（HTML/JSON）→ 展示给用户确认 → 用户确认后调用老系统的 API 写入。

**关键约束**：
- AI 只负责"填表单草稿"，不负责"提交"
- 高危操作（删除、退款、金额修改）必须二次确认
- 如果 AI 某些字段没提取到（比如折扣），表单里留空让用户手动补，而不是猜一个值

#### 1C. RAG 知识问答

**改造前**：新员工遇到客户问"XX 型号和 YY 型号有什么区别"，得翻 PDF 产品手册、问老同事、搜邮件、查知识库。

**改造后**：在对话框里问"XX 和 YY 的区别"，AI 从企业知识库里捞出相关产品文档段落，生成一句话对比 + 附原文链接。

**怎么做的**：把所有企业文档（产品手册、SOP、历史工单、FAQ）切片 → 向量化存到向量数据库 → 用户提问时检索最相关的文档片段 → 把片段 + 用户问题一起发给 AI 生成回答。

**关键约束**：
- 必须附原文链接，让用户可以溯源验证
- 当检索到的文档片段相关性低于阈值时，直接说"我暂时无法回答"而不是编一个
- 文档更新后要同步更新向量库

### 第二层：流程改造（价值更高但更重）

**一句话理解**：深入到具体业务流程里，用 AI 替代或增强原本需要人做判断、做审查的环节。这一层和第一层的区别是——第一层是"让操作变简单"，第二层是"让操作变聪明"。

#### 2A. 智能审批

**适用场景**：合同审批、报销审批、请假审批、供应商准入审批等需要人"看一眼"的流程。

**改造前**：一个报销单附了 5 张发票照片，审批人要一张一张点开看，判断发票内容和报销事由是否匹配、金额是否符合政策、是否有重复报销。一个审批平均耗时 5-10 分钟，审批量大了变成体力活。

**改造后**：AI 自动读取附件内容 → 比对审批规则 → 给出"建议通过/建议退回"及理由 → 审批人看一眼理由决定是否同意。

**怎么做的**：

```
员工提交报销单
       ↓
AI 预处理：
  1. OCR 识别发票金额、抬头、日期
  2. 对比报销政策和历史记录
  3. 检查是否存在重复发票号
  4. 检查金额是否超出该员工级别上限
       ↓
AI 输出审批建议 + 理由：
  "建议通过。3 项合规，1 项注意：餐费发票金额 825 元，
   略高于公司人均 150 元的日常标准，但发票备注注明为'客户招待'，
   符合商务接待政策。"
       ↓
审批人点"同意"或"退回"
```

**关键约束**：
- AI 只出建议，不出决定。审批责任永远在人
- AI 的每一条判断必须附带依据（"因为发票金额 X > 政策上限 Y"），让审批人能复核
- 如果有规则 AI 不确定（比如发票模糊 OCR 置信度低），必须标注出来提醒人工确认

#### 2B. 异常检测 + 自动处置

**适用场景**：财务对账差异、库存异常波动、生产设备数据偏离、日志里的异常告警。

**改造前**：财务月底对账，发现系统 A 和系统 B 差了 3 万块，然后用一周时间一条一条对明细找差异。IT 运维监控系统报了 200 条告警，其中 190 条是误报，真正有问题的那 10 条被淹没了。

**改造后**：AI 每天自动扫描关键指标，发现异常后给出"这是什么问题、影响多大、建议怎么处理"的日报。低风险操作可以自动执行。

**怎么做的**：

```
每日定时跑批：
  1. 拉取当日关键指标（交易额、退款率、库存周转、对账差异）
  2. AI 对比历史同期数据 + 业务规则，判断是否有异常
  3. 有异常 → AI 分析可能原因并给出处置建议
  4. 低风险操作（如发送对账异常通知、标记可疑交易）自动执行
  5. 高风险操作（如冲账、退款）生成待办工单由人处理
```

**关键约束**：
- 只自动执行"通知类"和"标记类"操作，涉及资金和库存变更的操作必须人执行
- 异常判断阈值由业务方确认，不能 AI 自己拍
- 每天只推送真正重要的异常（AI 做初筛），防止变成另一个"信息垃圾场"

#### 2C. 非结构化数据处理

**适用场景**：合同关键信息提取、客户聊天记录里的需求归类、邮件里的订单信息自动录入。

**改造前**：业务员收到客户发的合同扫描件（PDF），打开看一眼，手动把甲乙方、金额、付款条款、到期日敲进系统。一份合同 15 分钟，一天 10 份合同就 2.5 小时没了。

**改造后**：把合同 PDF 拖进系统，AI 自动提取所有关键字段填好表单，业务员核对一下有无偏差就行。一份合同缩短到 2-3 分钟。

**怎么做的**：

```
上传合同 PDF
  ↓
AI 解析 PDF → 提取文本 → 识别关键字段：
  ├─ 甲方：XX 有限公司
  ├─ 乙方：YY 科技有限公司
  ├─ 合同金额：500,000 元
  ├─ 付款方式：签约付 30%，验收后付 70%
  ├─ 生效日期：2026-05-01
  ├─ 到期日期：2027-04-30
  └─ 自动续约：是（到期前 30 天通知）
  ↓
自动填入合同管理系统的表单
  ↓
业务员核对 + 确认提交
```

**关键约束**：
- AI 提取的每个字段必须标注置信度，低置信度字段高亮提醒人工确认
- 金额、日期、甲方乙方名等关键字段即使置信度高也建议人工过一眼
- 提取结果要标注来源（从 PDF 第几页第几段提取的），方便溯源

### 第三层：系统缝补（ToB 需求极大）

**一句话理解**：传统企业普遍是烟囱系统——ERP 一套、CRM 一套、OA 又一套，数据不通全靠人力搬。这一层用 AI 做"系统数据搬运工"。

**为什么这是 AI 独特的优势**：传统 ESB/ETL（企业服务总线/数据抽取工具）也能做系统对接，但前提是两边的接口和数据格式都是标准的。实际情况是——系统 A 导出的是会计做的 Excel 模板（合并单元格、不规则表头），系统 B 的导入格式完全不同。传统集成方案需要人先手动做格式转换，AI 可以直接处理这种非标准的、需要"理解"的转换。

#### 3A. 跨系统数据搬运

**场景举例**：
- 电商订单在电商平台后台导出的是 Excel，要录入到企业自己的 ERP 里
- CRM 里的合同签约后，要在财务系统里创建应收账
- 仓库管理系统的入库数据，要同步到会计系统做库存账

**改造前**：运营每天从系统 A 导出 Excel → 手动调整格式 → 复制到系统 B 的导入模板 → 上传。每天 1-2 小时重复劳动。

**改造后**：AI 自动读取系统 A 的导出数据（不管什么格式），理解内容后按系统 B 的格式自动填好，生成导入文件或直接调用 API 写入。

**怎么做的**：

```
定时任务触发：
  1. 从系统 A 的数据接口/文件目录拉取最新数据
  2. AI 识别数据格式（Excel? CSV? 合并单元格? 不规则表头?）
  3. AI 理解数据内容（这条是"已付款订单"，金额是 199 元）
  4. AI 将内容映射到系统 B 的字段结构
  5. 生成系统 B 可接受的格式，写入或生成导入文件
  6. 记录转换日志，异常数据推送人工处理
```

#### 3B. 格式与语义转换

**场景举例**：
- 系统 A 的"已完成"等于系统 B 的"已完结"还是"已关闭"？
- 系统 A 的金额是元，系统 B 的金额是分
- 系统 A 的商品分类叫"3C 数码"，系统 B 叫"电子产品"

**传统做法的困境**：需要人建一张巨大的映射表，而且每次两边系统升级都可能新增或变更枚举值，映射表要持续维护。

**AI 的做法**：把这些映射规则用自然语言描述清楚，让 AI 做语义判断。遇到不确定的映射推给人确认，确认一次后记入规则库。

**关键约束**：
- 初次建立映射时，AI 的判断结果必须人工审核一遍
- 审核通过后，该映射存入确定性规则库，下次直接走规则不走 AI（保证确定性）
- 遇到规则库里没见过的值，再推给 AI 判断 + 人工确认
- 涉及资金和库存的同步，每条记录必须留痕可追溯

---

### 三层之间的关系

三层不是"做完一层才能做下一层"的瀑布关系，而是**可以并行、可以跳着做、但建议从第一层切入**：

- 第一层做出效果了（用户确实在用了），客户信任就有了，后续报价和合作更顺
- 第一层的自然语言查询能力可以直接复用到第二层（比如智能审批里的"查历史类似审批"）
- 第三层可以和第一层同时做——客户往往同时有"查询难"和"系统不通"两个痛

**建议落地顺序**：先做一个 2 周的套壳层 MVP（比如就做一个"自然语言查销售数据"的功能），内部跑通 → 收集用户反馈 → 决定下一步是深入哪个方向。

## 二、确保 AI 输出可靠性的五大手段

### 2.1 Schema 注入 + 示例数据

**核心思路**：把隐式的、只有老员工知道的业务知识，变成显式的、AI 能看到的约束条件。

每次调用 AI 生成 SQL 时，system prompt 中注入：

1. **相关表的完整 DDL**（字段名、类型、注释）
2. **每表 3-5 条示例数据**（暴露业务细节：金额单位、软删除、状态枚举等）

**示例**：

```
=== 相关表DDL ===
CREATE TABLE orders (
    id           BIGINT PRIMARY KEY,
    customer_id  BIGINT NOT NULL,
    total_cents  BIGINT NOT NULL,         -- 金额(分)
    status       VARCHAR(20) NOT NULL,     -- pending/paid/cancelled/refunded
    paid_at      TIMESTAMP,
    deleted_at   TIMESTAMP DEFAULT NULL    -- 软删除
);

=== 示例数据 ===
| id | customer_id | total_cents | status  | paid_at             | deleted_at |
|----|-------------|-------------|---------|---------------------|------------|
| 1  | 1           | 19900       | paid    | 2026-04-01 12:00:00 | NULL       |
| 2  | 2           | 50000       | paid    | 2026-04-02 15:00:00 | NULL       |
| 5  | 2           | 8800        | refunded| 2026-04-04 10:00:00 | NULL       |
```

**效果**：SQL 准确率从 60-70% 提升到 90%+。

**自动化**：写脚本从数据库自动抽取 DDL（SHOW CREATE TABLE）和示例数据（SELECT * LIMIT 5），无需手写。

---

### 2.2 模板化兜底（确定性执行）

**核心思路**：AI 不写完整 SQL，AI 只负责意图识别和参数提取。

```
预定义 50-80 个高频查询模板
用户问题 → AI 匹配模板 → 提取参数 → 填入模板
```

```
模板示例：
"最近N天XX商品的销售趋势" →
  SELECT date, SUM(total_cents)/100 AS amount
  FROM orders
  WHERE product_category = {category}
    AND paid_at >= {start_date}
    AND deleted_at IS NULL
  GROUP BY date
  ORDER BY date
```

**命中逻辑**：向量相似度匹配模板 → 匹配成功直接填参（100% 准确）→ 匹配失败才走 AI 生成 SQL 流程。

---

### 2.3 工程化质检（四层拦截）

质检规则来源于线上翻车案例的持续沉淀。

#### 第 0 层：语法检查（零成本）

```python
import sqlparse
parsed = sqlparse.parse(sql)
if not parsed:
    return False, "SQL语法解析失败"
```

#### 第 1 层：白名单约束（最实用）

- 只允许 SELECT
- 禁止访问未授权的表
- 禁止无 WHERE 的全表扫描
- 禁止 DROP / ALTER / INSERT / UPDATE / DELETE / EXEC 等危险关键字

#### 第 2 层：执行安全检查（在测试库用 EXPLAIN 跑）

- 预估扫描行数超过阈值（如 10 万行）→ 拒绝
- 检查是否使用索引
- 检测 filesort / temporary table

#### 第 3 层：语义规则校验（业务逻辑）

- 金额字段是否做了分转元处理
- 软删除字段（deleted_at）是否过滤
- 时间范围是否合理
- 状态枚举值是否正确

#### 质检失败的处理：重试循环

```
AI 生成 SQL → 四层质检 → 失败？→ 把失败原因喂回 AI → 重新生成（最多 3 轮）
                                                                    ↓
                                                            3 轮后仍失败 → 降级处理
```

**效果**：加上重试循环后，最终可用 SQL 从 ~75% 提升到 ~95%。

---

### 2.4 生成前规划（Planning）

**核心思路**：复杂查询不要直接生成 SQL，先让 AI 出执行计划，确认无误后再生成。

```
用户问题
  ↓
第 1 步：AI 先出执行计划（不写 SQL）
  "需要查 customers JOIN orders
   条件：paid_at 在上个月、status=paid、排除黑名单
   聚合：按 customer_id 分组，SUM(total_cents)
   排序：降序，取前 10"
  ↓
第 2 步：AI 根据计划生成 SQL
  ↓
第 3 步：质检层
```

**效果**：首轮通过率从 ~70% 提升到 ~85%。

---

### 2.5 生成与评审分离（多角色）

**核心思路**：不要用同一个 AI 既当考生又当考官。用独立的 AI 调用（甚至另一个模型）来评审生成结果。

```python
# 生成 SQL：用能力强但可能"偷懒"的模型
sql = claude.generate(user_question, schema)

# 评审 SQL：评审者不知道生成者的思考过程，独立审查
review = claude.review(sql, user_question, checklist=[
    "表名、字段名是否与 schema 一致",
    "金额是否做了分转元",
    "是否过滤了软删除",
    "时间条件是否正确",
    "聚合逻辑是否合理"
])
```

**效果**：多截住 15-20% 的问题，因为同一模型评审自己的输出有盲点。

---

## 三、进阶优化

### 3.1 动态 Schema 裁剪

不要把所有表的 DDL 都扔进 prompt。分两步：

```
第 1 步：AI 做表选择（轻量 prompt，只给表名列表 + 注释）
  → 选出 ["customers", "orders"]

第 2 步：只注入选中表的完整 DDL + 示例数据
  → 精度更高，token 消耗更低
```

### 3.2 历史案例库（动态 Few-shot）

每次成功生成且用户采纳后，存入案例库（问题文本 → 向量 → SQL）。下次生成时检索最相似的 3 个历史案例注入 prompt。

```python
similar_cases = vector_search(question, top_k=3)
prompt = f"""
    {schema_context}
    历史类似问题及对应SQL（已验证）：
    {format_cases(similar_cases)}
    用户新问题：{question}
"""
```

**优势**：案例是真实使用场景中验证过的，比手写示例数据更贴合实际。

## 四、降级策略

AI 不是 100% 可靠，必须有明确的降级链路：

```
优先级 1：模板匹配（100% 准确）
   ↓ 匹配不到
优先级 2：AI 生成 + 质检 + 重试 3 次（~95% 准确）
   ↓ 三次不过
优先级 3：拆解为子查询，逐个生成再合并
   ↓ 还是不行
优先级 4：友好提示 + 推荐人工查询入口
  "抱歉，您的问题暂时无法自动处理。
   已为您打开高级查询页面，相关条件已预填。"
```

**关键原则**：查询类 90% 准确就有价值（错了查不出东西，不造成损失）；写入类必须人确认后执行；决策类只做建议不做执行。

## 五、可观测性（最容易被忽视）

将所有中间步骤记录下来：

```python
log = {
    "trace_id": "uuid",
    "question": "上个月哪个客户消费最多",
    "planning": "...",
    "table_selection": ["customers", "orders"],
    "matched_template": None,
    "sql_attempts": [
        {"attempt": 1, "sql": "...", "errors": ["缺少deleted_at过滤"]},
        {"attempt": 2, "sql": "...", "errors": []},
    ],
    "final_sql": "SELECT ...",
    "user_feedback": "采纳",
    "latency_ms": 2400,
}
```

这些日志能回答三个关键问题：
- **哪个环节耗时最长？** → 优化点
- **哪种错误类型最常见？** → 加规则
- **用户最终采纳率是多少？** → 业务价值

---

## 六、这就是 Agentic Workflow

传统做法是"调一次 API 拿结果"。Agentic Workflow 是**把一次 LLM 调用变成一个可观测、可控制、可优化的流程**。

| 模式 | 对应实现 |
|------|----------|
| **Reflection（反思）** | AI 生成 SQL → 质检发现问题 → 喂回去重生成 |
| **Tool Use（工具调用）** | 质检工具（语法检查、EXPLAIN、语义规则） |
| **Routing（路由）** | 模板匹配 vs AI 生成的分发 |
| **Planning（规划）** | 先生成执行计划，再转 SQL |
| **Multi-agent（多智能体）** | 生成和评审使用独立的 AI 调用 |

---

## 七、对外包的商业价值

1. **工程化 = 溢价**：客户觉得"调个 API 就完事"时报价 5000 都嫌贵；你能说清楚 Schema 注入、四层质检、降级链路的设计，报 5 万也合理。

2. **方案难被白嫖**：模板设计、质检规则、语义层建模都是针对具体业务场景定制的，看流程图抄不走。

3. **谈方案时的安全策略**：
   - 谈思路，不交方案（大方向怎么说，具体技术选型不说）
   - 用"之前做过的类似项目"代替"针对你需求的方案"
   - 方案分阶段交付（签约前一页纸概要，签约后详细设计文档）
   - 报价单里用"实施计划/排期"代替技术实现细节

4. **质检规则沉淀**：先不加质检跑一周收集案例 → 人工 review 出错 case → 每个模式抽象为一条规则 → 加入质检层。一个月后规则基本饱和。

## 八、一句话总结

> 强提示词（Schema 注入 + 模板化）+ 工程化质检（四层拦截 + 重试循环）+ 生成前规划 + 评审分离 + 案例库 + 动态裁剪 + 降级链路 + 可观测性 = 可交付的 AI 工程化产品。

全部是工程代码，不依赖模型本身的能力提升。模型升级是额外收益，不升级也够用。
